# SPP Engine Vision — الرؤية الرسمية لمحرك SPP

> **المنتج الحقيقي هو Koil.** الملفات أدوات. Benchmark عدسة فحص — ليس منطق المحرك.

---

## الهدف النهائي

استقبال ملف **لم يره النظام من قبل**، ثم:

1. اكتشاف بنيته
2. فهمه كما يفهمه مدير عقارات محترف
3. بناء Property Knowledge
4. الاستنتاج (Koil Reasoning)
5. التوصيات العملية

**ليس** Parser متقدماً يعيد عرض Excel.

---

## مبدأان أساسيان

### 1. المحرك عام — ملفات المالك للتحقق فقط

- ملفات Golden Benchmark **ليست** قاعدة بيانات ولا نموذج تدريب.
- سؤال Benchmark الوحيد: *هل المحرك العام فهم هذه الملفات؟*
- أي إصلاح خاص بملفات مالك واحد = **فشل نضج المحرك**.
- كل إصلاح يجب أن يكون **قاعدة عامة** أو **تعلّم خاص بالعميل** (انظر الطبقة 3).

### 2. لا Rule Explosion

- لا نحل كل مشكلة بـ `if` أو Rule جديدة عامة.
- القواعد الثابتة للحقائق المالية المؤكدة فقط — ليس لكل شكل ملف.

## الطبقات الثلاث المتوازنة

```
أي ملف
    ↓
┌─────────────────────────────────────┐
│ 2. AI Understanding                 │  فهم النصوص، الأعمدة، العلاقات، الملاحظات
│    (لا حسابات)                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ 1. Deterministic Rules              │  حسابات، تواريخ، متأخرات، عقود، تحقق
│    (لا AI)                          │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Property Knowledge Engine           │  حقائق موحّدة عن العقار
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│ Koil Reasoning Engine               │  استنتاج + توصيات (قواعد من Knowledge)
└─────────────────────────────────────┘
    ↑
┌─────────────────────────────────────┐
│ 3. Learning Layer                   │  تصحيحات المستخدم → معرفة خاصة بالعميل
│    (ليست Rules عامة)                │
└─────────────────────────────────────┘
```

### الطبقة 1 — قواعد ثابتة (Deterministic)

**تُستخدم لـ:**
- الحسابات المالية
- التواريخ والفترات
- المتأخرات والتحصيل
- حالة العقود
- التحقق من اكتمال البيانات

**لا AI هنا.**

### الطبقة 2 — ذكاء اصطناعي للفهم (AI Understanding)

**تُستخدم لـ:**
- فهم النصوص والملاحظات
- فهم العلاقات (مستأجر ↔ وحدة ↔ عقد ↔ دفعة)
- اكتشاف معنى الملاحظات
- التعرف على أنواع البيانات والأعمدة
- اكتشاف الأنماط المختلفة بين العملاء

**لا AI للحسابات.**

### الطبقة 3 — التعلم المستمر (Learning Layer)

**عند تصحيح المستخدم مرة واحدة:**
- لا نضيف Rule عامة جديدة
- يُخزَّن النمط كـ **معرفة خاصة بالعميل** (Client Profile / Memory)
- أمثلة: مكان رقم العقد، الجوال في الملاحظات، صيغة الدفع

**الهدف:** كلما استُخدم النظام أكثر → أقل Rules، أكثر فهمًا، أكثر دقة.

## ما نرفضه

| مرفوض | السبب |
|--------|--------|
| Hardcode لملفات مالك واحد | يكسر العالمية |
| Benchmark يتحكم في منطق المحرك | Benchmark للقياس فقط |
| Rule جديدة لكل عميل جديد | Rule Explosion |
| AI يحسب المتأخرات أو الإيجار | الحسابات deterministic |
| نجاح يعتمد على Mock فقط | لا يثبت فهمًا حقيقيًا |

---

## معيار قبول أي تطوير

قبل دمج أي تغيير في المحرك:

1. هل يخدم **أي عميل** (أو طبقة التعلم الخاصة به) — وليس ملفًا واحدًا؟
2. هل هو في **الطبقة الصحيحة** (Rules / AI / Learning)؟
3. هل يزيد **Rule Explosion**؟ إن نعم — أعد التصميم.
4. هل يجعل **Koil أذكى** — وليس صفحة عرض أخرى؟
